# AGENTS.md

## Project Overview
This is a lightweight PHP starter website for CMaNGOS server operators who need basic account login and registration against a CMaNGOS `realmd` database. It uses procedural PHP, Composer for the `laizerox/php-wowemu-auth` SRP dependency, Bootstrap 5 assets/CDNs, and Apache-style rewrites.

## Environment Setup
1. Use PHP 8.2+ with the `mysqli` extension enabled and a web server such as Apache or Nginx.
2. Ensure a CMaNGOS `realmd` MySQL/MariaDB database is reachable.
3. Dependencies are committed in `vendor/`, but run `composer install` if you need to recreate them from `composer.lock`.
4. Update `configuration.php` with the local database host, username, password, database name, `GM_LEVEL`, and `EXPANSION_PACK`. There is no `.env.example`; configuration is currently PHP constants.
5. Serve from the repository root so `.htaccess` can rewrite `/login` to `login.php` and `/register` to `register.php`.

## Build & Run
- There is no compile/build step.
- Local PHP smoke server: `php -S 127.0.0.1:8000`
- Apache/Nginx deployment: point the document root at this directory and preserve the rewrite behavior in `.htaccess`.
- Refresh Composer dependencies only when needed: `composer install` or, for dependency upgrades, `composer update`.

## Code Style & Conventions
- No formatter or linter is configured; match the existing procedural PHP style and indentation.
- Keep shared layout in `header.php` and `footer.php`; keep page-specific content in `index.php`.
- Use `include __DIR__ . '/file.php'` and `require_once __DIR__ . '/vendor/autoload.php'` for local includes/autoloading.
- Keep database/auth defaults centralized in `configuration.php`; do not hardcode those values in handlers.
- Do not edit `vendor/`; change Composer constraints in `composer.json` and refresh `composer.lock` when dependency changes are required.
- Preserve Bootstrap-first markup and the existing `assets/css/custom.css` override pattern for UI changes.
- Prefer clear, small procedural helpers over introducing a framework or broad OO structure.

## Architecture Notes
- `header.php` starts the PHP session and renders the document head/navigation; `footer.php` renders forms for the login and register modals.
- Auth form submissions post to extensionless routes (`/login`, `/register`) that depend on `.htaccess` rewriting to the matching PHP files.
- `login.php` reads the account salt/verifier from `realmd.account` and compares it with a verifier generated by `Laizerox\Wowemu\SRP\UserClient`.
- `register.php` creates SRP salt/verifier values through `UserClient` and inserts a new account row using configured GM level and expansion values.
- Business logic currently lives directly in the handler files. Keep changes tightly scoped unless a task explicitly asks for a refactor.
- The database integration assumes the CMaNGOS `realmd` schema, especially the `account` table and columns used by `login.php` and `register.php`.

## Commit & PR Conventions
- No project-specific branch naming or commit message convention is documented; use short, imperative commit messages that describe the changed behavior.
- Keep PRs focused and include manual validation notes for login, registration, configuration, and UI changes.
- Before opening a PR, check `git status --short`, run `php -l` on changed PHP files, and avoid staging unrelated local files.

## Things to Avoid
- Do not introduce raw SQL interpolation for user input in new or modified code; use prepared statements.
- Do not commit real database credentials or server-specific secrets in `configuration.php`.
- Do not assume the committed `vendor/` directory means dependencies should be edited in place.
- Do not break extensionless `/login` and `/register` routes when changing forms or server config.
- Do not add a framework, routing layer, asset pipeline, or build step unless the task explicitly calls for it.
- Do not leave auth flows with blank-page success/failure behavior when touching login or registration handling.
